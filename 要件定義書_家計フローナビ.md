# 家計フロー・ナビ 要件定義書

| 項目 | 内容 |
| --- | --- |
| プロダクト名 | 家計フロー・ナビ（家族共有 家計簿PWA） |
| 版 | 1.1 |
| 作成日 | 2026-06-03 |
| 利用者 | 夫婦2人のみ（プライベート利用） |
| 形態 | PWA（Progressive Web App） |
| 費用方針 | 完全無料（無料枠内で運用） |

---

## 1. 目的・背景

夫婦2人で家計を共有・管理するための、スマホ中心の家計簿アプリを作る。
すでに動作する試作品（HTML + Firebase Firestore + Chart.js）が存在し、本書はそれを土台に
**レシート読み込み（明細化）** と **購入履歴に基づく価格比較** を加え、
**PWA化** と **2人限定の認証・セキュリティ** を整えるための要件をまとめたものである。

### 1.1 解決したい課題
- 支出の記録を夫婦どちらからでも入力でき、リアルタイムに共有したい。
- レシートを撮るだけで、商品ごとの明細を記録できるようにしたい。
- 「同じ商品なら、自分たちが過去に買った中でどの店が安かったか」を知りたい。
- 追加サービスの登録や月額費用は発生させたくない（完全無料）。

---

## 2. 利用者・利用環境

### 2.1 利用者
- 本人（夫）と配偶者（妻）の **2名のみ**。
- 両者は同一のデータを共有し、双方が閲覧・入力・編集・削除を行える（権限差なし）。

### 2.2 利用環境
- 主にスマートフォン（iOS / Android）のブラウザおよびホーム画面に追加したPWA。
- 補助的にPC（デスクトップブラウザ）からも利用可能。
- 通信前提（オフライン時はFirestoreのキャッシュとPWAのアプリ本体キャッシュで起動・閲覧は可能、書き込みは復帰時に同期）。

---

## 3. 全体方針・技術スタック（確定）

検討の結果、外部サービスを増やさず **Firebase 1本に集約** する構成を採用する。

| レイヤー | 採用技術 | 費用 | 補足 |
| --- | --- | --- | --- |
| フロントエンド | HTML + バニラJS（試作品を踏襲） | 無料 | 既存デザイン・機能を維持 |
| ホスティング | Firebase Hosting | 無料 | HTTPS標準対応（PWA要件を満たす） |
| データベース | Firebase Firestore | 無料枠 | リアルタイム同期。**無操作による一時停止なし** |
| 認証 | Firebase Authentication（Googleログイン） | 無料 | 2人のアカウントに限定 |
| グラフ | Chart.js（CDN） | 無料 | 既存踏襲 |
| OCR | Tesseract.js（端末内処理） | 無料 | **APIキー・中継サーバー不要** |
| アプリ形態 | PWA（manifest + Service Worker） | 無料 | インストール・全画面・オフライン起動 |

### 3.1 不採用とした選択肢と理由
- **Gemini API**: 利用できない環境のため不採用。OCRは端末内のTesseract.jsに切り替え。
- **Supabase**: 既にプロジェクト2つを作成済みで無料枠（同時2プロジェクト）に空きがないため不採用。また無料枠は7日間の無操作でプロジェクトが一時停止する点も家計簿用途に不向き。
- **Vercel**: APIキーを隠す中継関数が必要な場合の置き場所として検討したが、OCRを端末内処理にしたため中継が不要となり、Firebase Hostingに集約できるため不採用。
- **認証なし運用**: Firestoreが実質「誰でも読み書き可」となり、家計データが第三者に閲覧・改ざんされうるため不採用。

## 4. 機能要件

### 4.1 既存機能（維持する）
試作品が備える以下の機能は、原則そのまま維持する。

- 支出の新規記入・編集・削除（用途カテゴリ／日付／金額／購入場所／メモ）
- ダッシュボード：当月支出サマリー、用途別タグ表示
- グラフ表示：用途別支出（円グラフ等）、月別支出推移（棒／折れ線／円の切替）
- 履歴一覧：月・用途でのフィルタリング、編集・削除導線
- 簡易アドバイス（ルールベース）：先月比のペース判定、最大支出カテゴリの指摘
- 来月予測：月別合計の直線回帰による概算
- リアルタイム同期（Firestore onSnapshot）

### 4.2 追加機能

#### F-1. 認証・2人限定アクセス（最優先）
- Googleログインを必須とし、未ログイン時はアプリ本体を表示しない（ログイン画面のみ）。
- アクセスを **特定の2つのアカウント（UID）に限定** する（Firestoreセキュリティルールで強制）。
- ログイン中のアカウント表示、ログアウト操作を提供する。
- 初回セットアップ補助として、ログイン後に自身のUIDを画面に表示し、ルールへ登録しやすくする。

#### F-2. 商品マスタ管理
- 夫婦が **自分たちで商品マスタを登録・編集・管理** する（名寄せは手動でマスタに集約する方針）。
- 商品マスタの項目：商品名、カテゴリ、メモ（任意）。
- レシート明細やマニュアル入力の各行を、商品マスタの商品に紐付けて記録する。
- マスタに無い商品はその場で新規登録できる。

#### F-3. 明細付き購入記録
- 1回の買い物（＝レシート1枚相当）を「購入記録」として保存する。
- 購入記録の項目：日付、店名、合計金額、メモ（任意）、明細（複数行）。
- 明細の各行：紐付け商品（商品マスタ）、表示名、価格、数量。
- 既存の「支出記入」も引き続き利用でき、明細管理と併存させる。

#### F-4. レシート読み込み（OCR・半自動）
- スマホでレシートを撮影、または画像を選択して読み込む。
- Tesseract.js（端末内）で **品目テキスト＋金額の行を抽出し「下書き」** を生成する。
- 抽出結果は **確認・手直し画面** で編集できる（誤読の修正・行の追加削除）。
- 各行を商品マスタに紐付けて、購入記録として保存する。
- **全自動ではなく「OCRで下書き → 人が確認」の半自動** を前提とする（後述のOCR精度の制約に基づく）。

#### F-5. 価格比較（最安店・購入履歴ベース）
- 比較範囲は **自分たちの購入履歴の中のみ**（外部・市場価格は対象外）。
- 比較単位は **1点あたりの価格**（内容量あたりの単価換算は対象外）。
- 商品ごとに、過去の購入履歴を店別に集計し、**最も安かった店と価格** を表示する。
- 補助情報として、店別の最新価格や平均価格を表示してもよい（任意）。

#### F-6. PWA
- ホーム画面に追加（インストール）できる。
- 全画面（standalone）表示、アプリアイコン、スプラッシュ表示。
- Service Workerでアプリ本体（HTML/CSS/JS）をキャッシュし、オフラインでも起動できる。

### 4.3 段階リリース計画（実装順）
価値の高い順に、段階的にリリースする。各フェーズ単体でも実用になることを重視する。

| フェーズ | 内容 | 対応機能 |
| --- | --- | --- |
| **1** | 認証＋2人限定セキュリティルール ＋ PWA土台 | F-1, F-6 |
| **2** | 商品マスタ ＋ 明細付き購入記録 ＋ 商品ごとの最安店ビュー | F-2, F-3, F-5 |
| **3** | レシートOCR（Tesseract.js・半自動） | F-4 |
| **4**（任意） | 既存のアドバイス・予測を店別／明細データで強化 | 4.1の拡張 |

> フェーズ3が難航しても、フェーズ1・2のみで「明細管理＋価格比較」は成立する。

### 4.4 UI/UX（動的アニメーション）要件
操作が気持ちよく、状態の変化が直感的に伝わるよう、UIに動的なアニメーションを加える。
全フェーズ共通の横断要件として実装する（特定フェーズ専用ではない）。

- **画面遷移**: ダッシュボード／カテゴリ／入力／確認などの切替にフェード・スライドのトランジションを付ける（試作品のfadeInを発展）。
- **数値のカウントアップ**: 当月支出や合計金額などの主要数値を、0から実数まで滑らかに増加表示する。
- **グラフの描画アニメーション**: Chart.jsの描画時アニメーション（棒の伸び・折れ線のドロー・円の回転）を有効にする。
- **マイクロインタラクション**: ボタンの押下・ホバー、保存成功時の軽いフィードバック（チェックの出現等）。
- **リストの出現演出**: 履歴・明細の各行が順次フェードイン／スライドインする（スタガー表示）。
- **ローディング表現**: OCR処理中・同期中など、待ち時間に進捗が伝わるアニメーション（スピナー／プログレス／スケルトン）。
- **配慮事項（非機能）**:
  - OSの「視差効果を減らす」設定（`prefers-reduced-motion`）を尊重し、その場合はアニメーションを控えめ／無効にする。
  - モバイルでもカクつかないよう、`transform`／`opacity`中心の軽量な実装とし、パフォーマンスを優先する。
  - 過度に長い・派手すぎる演出は避け、操作の邪魔をしない範囲にとどめる。

## 5. データモデル（Firestore・案）

ルートは家族単位のドキュメント `households/{householdId}` 配下にまとめる。
（`householdId` は現行の `oshiro_family` を踏襲。）

```
households/{householdId}/
├─ entries/{entryId}        // 既存：単発の支出記録（互換維持）
│    ├─ category : string   // 用途カテゴリ
│    ├─ date     : string   // YYYY-MM-DD
│    ├─ amount   : number   // 金額（円）
│    ├─ place    : string   // 購入場所
│    ├─ memo     : string
│    └─ createdAt: timestamp
│
├─ products/{productId}     // 商品マスタ（手動管理）
│    ├─ name     : string   // 商品名
│    ├─ category : string   // カテゴリ
│    ├─ note     : string   // メモ（任意）
│    └─ createdAt: timestamp
│
└─ purchases/{purchaseId}   // 明細付き購入記録（レシート1枚相当）
     ├─ date     : string   // YYYY-MM-DD
     ├─ store    : string   // 店名
     ├─ total    : number   // 合計金額
     ├─ memo     : string
     ├─ items    : array     // 明細
     │    └─ { productId, name, price, qty, category }
     └─ createdAt: timestamp
```

- 価格比較は `purchases.items` を商品（`productId`）と店（`store`）で集計して算出する。
- 既存の集計・グラフは `entries` と `purchases` の双方を合算対象にできるよう拡張する（フェーズ2で調整）。

## 6. 非機能要件

| 区分 | 要件 |
| --- | --- |
| コスト | すべて無料枠内で運用。有料サービス・課金は使用しない。 |
| プライバシー | データへのアクセスは指定した2つのUIDのみに限定する。 |
| セキュリティ | Firestoreセキュリティルールで認証必須＋UID許可リストを強制する。 |
| 可用性 | Firestore無料枠は無操作による一時停止がないため、たまの利用でも即時利用可能。 |
| オフライン | Firestoreのオフライン永続化＋Service Workerのキャッシュで、起動・閲覧を可能にする。 |
| 対応環境 | モバイル優先のレスポンシブ。最新のモバイル／デスクトップブラウザを対象。 |
| 性能 | OCR処理は端末内で実行。日本語学習データの読み込みを考慮し、初回のみ時間がかかる旨をUIで明示。 |
| UI/UX | 動的アニメーションを付与。`prefers-reduced-motion` を尊重し、`transform`/`opacity` 中心の軽量実装でモバイル性能を確保する（4.4参照）。 |
| 保守性 | 既存のバニラJS構成・デザインを踏襲し、依存サービスを増やさない。 |

## 7. セキュリティ方針（2人限定）

- **認証**：Firebase Authentication の Google プロバイダを有効化する。
- **許可リスト方式**：セキュリティルールに、許可する2名のUIDを直接記載し、
  `request.auth.uid` がそのいずれかである場合のみ `households/{householdId}` 配下の読み書きを許可する。
- **UIDの確定手順**：
  1. アプリをデプロイし、各自が一度Googleログインする。
  2. アプリ画面に表示される自分のUIDを控える。
  3. 2名分のUIDをセキュリティルールに記載して再デプロイする。
- Firebase設定値（apiKey等）はクライアントに露出するが、セキュリティはルールで担保する（Firebaseの標準的な考え方に準拠）。

## 8. 制約・前提・既知のリスク

- **OCR精度の制約**：日本のレシートは感熱紙かつ店ごとにレイアウトが異なり、Tesseract.jsでの明細の自動抽出は取りこぼしが起こりうる。このため「OCRは下書き、人が確認・修正」という半自動運用を前提とする。
- **Google Auth設定**：FirebaseコンソールでのGoogleプロバイダ有効化、および承認済みドメインの設定が事前に必要。
- **無料枠の範囲**：Firestoreの無料枠（読み書き回数・保存容量）に収まる前提。2人の家計簿用途では十分とみなす。
- **iOSのPWA制約**：iPhoneでもホーム画面追加で利用可能だが、プッシュ通知等はAndroidより制限がある。本アプリの用途（撮影・記録・閲覧）では支障なし。

## 9. スコープ外（今回は作らない）

- ネット上の市場価格・他店チラシ等、外部データとの価格比較。
- 内容量あたりの単価（100gあたり等）への換算比較。
- 3人以上の利用、利用者ごとの権限差。
- 銀行口座・クレジットカードの**自動連携**（アカウントアグリゲーション）。
  - 理由：日本のアグリゲーションAPI（Moneytree LINK／Money Forward 等）はいずれも法人契約前提のB2B提供で、個人が無料で利用できる窓口がなく、トークン管理用のサーバーも必須となり「完全無料・サーバーなし」の方針に反するため。
- プッシュ通知（iOSの制約も踏まえ対象外）。
- 有料サービスの利用全般。

> **将来オプション（無料で実現可能な代替）**：カードの利用データを取り込みたい場合は、カード会社の会員サイトからダウンロードした**利用明細CSVの取り込み機能**が現実的（契約・サーバー不要・無料）。必要になった時点で別フェーズとして検討する。

## 10. 用語

| 用語 | 意味 |
| --- | --- |
| PWA | ブラウザ技術で作る、インストール可能・オフライン対応のWebアプリ。 |
| OCR | 画像から文字を読み取る技術。本書では端末内のTesseract.jsを指す。 |
| 名寄せ | 表記の異なる同一商品を1つにまとめること。本アプリでは手動の商品マスタで行う。 |
| UID | 各ユーザーをFirebase Authが一意に識別するID。 |
| 商品マスタ | 価格比較の基準となる、ユーザー管理の商品一覧。 |
| 購入記録 | 1回の買い物（レシート1枚相当）の明細付き記録。 |
